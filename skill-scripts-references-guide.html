An Anatomy · · · Recto / Folio 01

Skill 不是更长的提示词，
而是 AI 可以调度的工具包。

这份手册把 skill 设计里的 SKILL.md、references/ 和 scripts/ 拆开讲，并用当前仓库里的 PDF skill 作为样本，说明为什么关键流程应该脚本化——以及为什么这件事比再写一段更聪明的提示词重要得多。

“让 AI 判断的留在 SKILL.md，让 AI 查阅的放进 references/，让机器执行的放进 scripts/。” — 一句话判断标准
01
ChapterFoundations

核心判断

一个好的 skill 不应该只有 SKILL.md。入口文件负责告诉 Codex 什么时候触发、按什么流程做、去哪里找资源；references/ 保存不该反复塞进主说明里的领域知识；scripts/ 把高频、脆弱、可验证的动作变成稳定工具。

一句话判断：让 AI 判断的留在 SKILL.md，让 AI 查阅的放进 references/，让机器执行的放进 scripts/。
02
ChapterArchitecture

三层职责

入口 · ENTRY

SKILL.md

负责触发条件、总流程、资源导航和关键约束。它的质量标准不是信息最多，而是让另一个 Codex 快速知道下一步该做什么。

知识 · LORE

references/

保存 API 文档、schema、业务规则、详细流程、错误码和大量示例。只有相关分支需要时才读，避免污染主上下文。

执行 · ACT

scripts/

保存可执行工具，把文件解析、格式转换、API 调用、校验、渲染、写入等稳定动作从模型自由发挥中拿出来。

SKILL.md 应该包含

  • 触发条件：用户说什么、出现什么文件、遇到什么场景时要用这个 skill。
  • 总流程：先做什么，再做什么，遇到分支怎么判断。
  • 资源导航：什么情况下读取哪个 reference，什么情况下运行哪个 script。
  • 关键约束：必须遵守的顺序、禁止事项、验证要求。

references/ 适合存放

  • API 文档、CLI 参数说明、SDK 用法。
  • 数据库 schema、字段含义、业务口径。
  • 公司政策、品牌规范、合同模板说明。
  • 复杂流程的详细分支说明。
  • 大量示例、边界案例、错误码解释。

scripts/ 适合存放

  • 文件解析、格式转换、批量处理。
  • schema 校验、字段检查、数据清洗。
  • API 调用封装、分页拉取、重试逻辑。
  • 报表生成、图片/PDF/表格渲染。
  • 环境检查、权限检查、依赖诊断。
  • 任何 AI 容易写错、但程序可以稳定完成的步骤。

ChapterDecision Matrix

设计决策表

内容类型 放哪里 原因
触发条件 SKILL.md frontmatter description Codex 只有先看到 description 才知道要不要加载 skill。
主流程 SKILL.md 每次触发都需要知道。
分支导航 SKILL.md 决定后续读哪个文件或跑哪个脚本。
详细 API 文档 references/ 只有相关任务才需要。

校验逻辑 scripts/ 程序比自然语言稳定。
模板文件 assets/ 被复制或用于产出，不是给 Codex 阅读。
调试手册 references/troubleshooting.md 出错时再加载。
04
ChapterAudit

评审清单

评审一个 skill 是否还停留在“提示词阶段”，可以用下面的问题快速判断。

需要 references 的信号
  • SKILL.md 超过很多屏，混入大量资料。
  • 存在很多字段、规则、API、错误码，但没有独立 reference。
  • 同一个 skill 支持多个领域、平台、文件格式或工作流。
  • 失败后只能让 AI 猜，而没有 troubleshooting reference。
需要 scripts 的信号
  • 出现“复制下面代码运行”，但没有脚本文件。
  • 每次调用都需要重新写解析、转换、校验代码。
  • 存在稳定输入输出，却仍然交给 AI 自由发挥。
  • 缺少 inspect、transform、validate 这类确定性步骤。
05
ChapterCase Study

PDF 样本观察

当前仓库里的 pdf/ skill 不是一个“只有提示词的 skill”。它的 SKILL.md 负责给出 PDF 处理的总入口和常见能力说明；当任务进入“填写 PDF 表单”这个复杂分支时，SKILL.md 要求读取 forms.md，而 forms.md 再把流程拆成一系列脚本化步骤。

这个 skill 的设计思路：AI 负责判断、理解和决策；脚本负责检查、抽取、转换、校验和写入。

PDF 表单填写特别适合这样拆，因为它同时涉及可填字段识别、字段 ID、页码、合法选项、checkbox/radio 状态、PDF 坐标系和图片坐标系转换，以及写入后的重叠和偏位检查。

ChapterPipeline

脚本职责地图

1STEP识别表单类型
2STEP抽取可操作结构
3STEP决定字段含义
4STEP写入前校验
5STEP填充并验证
流程阶段 脚本 承担职责
识别表单类型 check_fillable_fields.py 判断 PDF 是否有可填表单字段，决定走 fillable 还是 non-fillable 分支。
抽取 fillable 字段结构 extract_form_field_info.py 提取字段 ID、页码、坐标、字段类型、checkbox/radio/choice 可用值。
视觉理解辅助 convert_pdf_to_images.py 把 PDF 渲染成 PNG，供 AI 或人判断字段在页面上的真实含义。
抽取 non-fillable 页面结构 extract_form_structure.py 从不可填 PDF 中提取文字标签、横线、checkbox、小矩形和行边界。
写入前校验 check_bounding_boxes.py 检查 label/entry 框是否相交、输入框高度是否小于字体大小。
视觉校验辅助 create_validation_image.py 在页面图上画出 label 和 entry 框，帮助检查坐标准不准。
填充 fillable 表单 fill_fillable_fields.py 根据字段 ID 写入真实 PDF 表单字段，并校验字段 ID、页码和值是否合法。
填充 non-fillable 表单 fill_pdf_form_with_annotations.py 用 FreeText annotation 把文本写到指定坐标，自动处理 PDF/图片坐标转换。
07
ChapterCatalogue

逐个脚本拆解

8 个脚本不必逐个细讲——课堂上只展开 2 个。这两个一起讲，就能讲清整个 skill 的设计哲学：凡是确定性问题，交给程序；凡是判断和理解，交给 AI。其余 6 个的角色列在本章末尾，同事课后自取。

check_fillable_fields.py流程分流器 · 一个 Boolean 决定半条流水线

这个脚本用 pypdf.PdfReader(...).get_fields() 判断 PDF 是否存在可填字段，决定后续走 AcroForm 字段填充流程，还是走视觉定位加 annotation 填写流程。

为什么不让 AI 判断：这是个二分判断——是或否，没有灰色地带。如果 AI 凭页面外观判断，可能把可填字段当成普通文本框走错分支；一旦分错，后续 6 个步骤全部失效。分流是 skill 中"最贵的判断"，必须钉死。

check_bounding_boxes.py写入前的几何安全门

它检查 label 框和 entry 框是否相交、不同字段的框是否互相重叠、entry 框高度是否小于字体大小。

为什么不让 AI 自检：几何重叠是确定性数学问题，但 AI 自己看 JSON 数组很容易看漏——它会在一长串坐标里"扫一眼觉得没问题"就放行。提前失败比生成错位 PDF 后返工便宜得多。这一类"自动驾驶到悬崖前先刹车"的脚本，是 skill 工具化最容易被忽视的一档。

其余 6 个 — 课后自查

同样遵循"AI 容易错的地方让程序兜底"的逻辑，课堂上不展开。列在这里供同事课后读 skill 时对照。

脚本 角色 为什么交给脚本
extract_form_field_info.py 抽取可填字段为 JSON 父子层级字段 ID 和 checkbox/radio 合法值不是肉眼能稳定判断的。
convert_pdf_to_images.py 给 AI 一个视觉判断面 渲染参数和图片命名需要稳定，否则后续坐标判断会漂移。
extract_form_structure.py 不可填 PDF 的结构雷达 坐标抽取是机械任务，AI 凭视觉猜坐标必偏。
create_validation_image.py 把坐标画到页面图 不应该每次让 AI 临时写 PIL 绘图代码。
fill_fillable_fields.py fillable 表单写入器 字段 ID 写错会静默失败，脚本能提前报错。
fill_pdf_form_with_annotations.py non-fillable 坐标写入器 PDF 与图片的 y 轴方向相反，坐标转换最易错。

09
ChapterTypes & Tells

脚本到底有哪几种

这一章不教写代码——它教同事怎么在自己的 SKILL.md 里把脚本"认出来"。类型是关键，语言只是落地时的选择：可以是 .py、.ps1、.sh、.js，也可以是 .sql 或一个编译好的 .exe，Codex 只关心它能跑、有明确输入输出。

核心模型：所有可以变成脚本的事情，都能归到五类——检查、抽取、转换、校验、执行。同事的 SKILL.md 里某段话能对应到其中一类，就找到了一个脚本机会。
— Languages — 同一类脚本可以用任何语言实现；下面是常见选择和它们各自擅长的场景。
.py Python 跨平台、生态最广。处理 PDF / Excel / OCR / 调 API / 数据科学几乎是默认。
.ps1 PowerShell Windows 原生。访问注册表、AD、Office、SharePoint、Teams 最顺，无需额外环境。
.sh Bash Linux / macOS 原生，Windows 上靠 Git Bash 或 WSL。处理文件、管道、Git、CI 流水线。
.js .ts Node.js 需要 npm 生态时：调云服务 SDK、操作浏览器（Playwright）、前端工具链。
.sql SQL 脚本 直接连数据库查询或写入。比让 AI 临时拼 SQL 更安全，能版本化、能 dry-run。
.bat .ahk Batch / AutoHotkey Windows 老式 cmd 调用，或桌面自动化（键鼠、窗口、剪贴板）。
.exe Go / Rust / .NET 编译产物 极速、单文件分发，跨机器跑无依赖。给非开发者用或性能敏感时选。
Before提示词阶段 
SKILL.md（30 屏）
├─ 触发条件
├─ "请先检查 Excel 是否存在"
├─ "把 PDF 里的表格读出来"
├─ "转换成 Markdown 然后..."
├─ "确保金额大于 0"
└─ "最后发邮件通知客户"
       (全是自然语言，全靠 AI 自己写)
After识别出五类脚本 
SKILL.md（10 屏）
├─ 触发条件 / 主流程
└─ scripts/
    ├─ I.   check_env.sh         检查类
    ├─ II.  extract_tables.py    抽取类
    ├─ III. csv_to_excel.py      转换类
    ├─ IV.  validate.ps1         校验类
    └─ V.   send_email.py        执行类
           (语言可混搭，按场景挑)
I 检查类 Inspect / Check

在动手之前，先看清楚现状。

SKILL.md 里这种话就是信号

  • 请先确认 Excel 文件存在
  • 如果用户装了 Python 3.10 以上...
  • 目录里如果有 .csv 文件...
  • 看看接口是否能连

常见例子

  • 工具 / 版本是否就位
  • 目录里有几个什么文件
  • API 是否可达、Token 是否有效
  • 环境变量 / 配置文件是否齐全
In 环境 / 文件 / 配置 → Out 现状清单 (JSON / 退出码)
为什么
是脚本 判断"装没装、有没有"是确定性问题。让 AI 凭直觉假设容易翻车——它会自信地继续往下走，错误堆到流程末尾才暴露。
常用 .sh .ps1 .bat .py
II 抽取类 Extract

从乱糟糟的原始材料里捞出结构化信息。

SKILL.md 里这种话就是信号

  • 请从这份合同里找出甲方乙方
  • 把 PDF 里的表格读出来
  • 扫描每个 .md 文件的 frontmatter
  • 识别这张截图里的金额

常见例子

  • PDF / Word / Excel 提取文字或表格
  • 仓库里扫 TODO / 依赖 / 死代码
  • 截图 OCR、发票识别、票据解析
  • 日志解析、关键字段抽取
In 原始文件 / 图片 / 网页 → Out 整齐 JSON
为什么
是脚本 抽取规则一旦写好就稳定；AI 每次重抽容易遗漏字段、字段名漂移、同一份合同两次产出不同结果。
常用 .py .js .ts .sh (grep / awk)
III 转换类 Transform / Convert

从一种格式变到另一种格式，信息守恒。

SKILL.md 里这种话就是信号

  • 把这份 CSV 转成 Excel
  • Markdown 渲染成 PDF
  • PDF 坐标换算成图片坐标
  • 把客户名单拆成中英文两份

常见例子

  • CSV / JSON / Excel / Parquet 互转
  • 模板 + 数据 → Word / PDF / HTML
  • 单位 / 时区 / 坐标系 / 货币换算
  • 文本规范化（全角半角、大小写）
In A 格式 → Out B 格式
为什么
是脚本 编码、分隔符、转义这些细节模型经常错；同一个转换不应该每次重造轮子——每次重造就意味着每次返工。
常用 .py .js .ts .ps1
IV 校验类 Validate

在真正写入或调用之前，先卡住明显的错误。

SKILL.md 里这种话就是信号

  • 确保必填字段都填了再提交
  • 金额必须大于 0
  • 客户编号不能重复
  • frontmatter 不能缺 description

常见例子

  • 必填字段、类型、范围校验
  • 业务规则（订单总价 = 单价 × 数量）
  • 唯一性 / 重复性检查
  • 文件命名 / 目录结构规范
In 待提交数据 → Out OK 或一份"错在哪"清单
为什么
是脚本 让 AI 自检相当于自己批改作业，常会睁一只眼闭一只眼；脚本失败抛错比生成错文件后返工便宜得多。
常用 .py .ts .ps1 .sh
V 执行类 Execute / Act

真正改动外部世界的动作——这一步出错代价最高。

SKILL.md 里这种话就是信号

  • 请发邮件通知客户
  • 在 Jira 创建 ticket
  • 更新数据库这条记录
  • 把文件保存到 OneDrive

常见例子

  • 发邮件 / Teams / 钉钉 / Slack
  • 调 API：创建 issue、上传文件、写数据库
  • 提交代码、推 PR、触发部署
  • 生成最终 Word / PDF / Excel 落盘
In 准备好的数据 → Out 外部世界发生改变（不可逆）
为什么
是脚本 重复发邮件、写错数据库都是不可逆事故。脚本能加幂等检查、权限检查、统一日志，AI 凭自然语言做不到。
常用 .py .ps1 .sh .sql .js .ts
— 怎么和同事讲这件事 —
  1. 让同事把 SKILL.md 大声读一遍，遇到 "请检查 / 请确认" 就标 I，"请提取 / 找出" 标 II，"请转成 / 渲染" 标 III，"必须满足 / 不能" 标 IV，"请发送 / 创建" 标 V。
  2. 标出来的每一处都是一个脚本机会。同事自己不写代码也没关系——先让标签出现在 SKILL.md 边上，至少知道"这里以后该是个脚本"。
  3. 挑标签最多的那一类先动手；找一位会写代码的同事或直接让 Codex 把它实现成脚本——不限定语言，谁顺手用谁。
10
ChapterCoda

给团队的启发

  1. SKILL.md 不负责承载所有细节，只负责入口、主流程和资源导航。
  2. 复杂分支放到 reference，例如 PDF skill 里的 forms.md。
  3. 每个关键流程节点都问一句：这里是让 AI 判断更合适，还是让脚本执行更合适？
  4. 只要是检查、抽取、转换、校验、写入，就优先考虑脚本。
  5. 脚本之间最好形成中间产物契约，例如 field_info.json、fields.json、field_values.json。
最终心智变化：skill 不是一份更长的提示词，而是一套 AI 可以调度的稳定工具链。
