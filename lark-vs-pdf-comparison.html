两个 skill,
同一条 路.

把 lark-bugs-assigned-to-me 放在 pdf 旁边比对, 是为了让"还差什么"显形. pdf 已经把"AI 判断 / 文档查阅 / 脚本执行"分到三层; lark 还在用一份 15KB 的提示词承担全部. 这份对照不是评判, 是诊断 — 它要回答的是: 下一步把哪一块先抽出来, 才最划算.

一个 skill 的稳定度, 取决于它把多少机械动作交给了脚本, 而不是交给了模型的记忆. — 拆解 03 / 复盘笔记

停一下 · Exercise 01 ≈ 10 MIN · 写下来再继续

看完两份目录, 在我开始拆解之前 —

下面三个问题都没有标准答案. 重要的是把答案写在纸上, 等会儿讲"同一类决定, 两种载体"时, 你的答案会自动和卡片对照, 不一致的地方就是收获.

  1. 这两份 skill 都在解决什么共同问题? 写出至少 3 条它们的相似处 — 触发条件、流程节点、产出形式, 任何角度都行.
  2. 假设只看目录结构, 不看内容 — PDF skill 已经做对了哪几件事是 lark 还没做的? 答案应该至少包含两个文件夹或文件类型.
  3. 如果今天就要动 lark 的目录, 你会先加哪一类文件或文件夹? 为什么是这一类, 而不是别的?
讲师提示这一题的目的是让大家在"看到答案之前先长出自己的判断". 不要急着翻页 — 给他们 8-10 分钟写, 然后请 2-3 位同事念出自己的 Q2 答案, 再进入下一章.

Chapter 02

同一类决定, 两种载体

两个 skill 在做的事可以一一对位: 都要分流、都要抽字段、都要拿数据、都要在写入前自查、都要产出稳定结果. 差别只在"这一步交给谁完成". 下面五张卡, 左 = pdf, 右 = lark.

01

流程分流

Branch decision
PDF

第一步必须运行 scripts/check_fillable_fields.py, 返回布尔判断, 决定走 AcroForm 还是 annotation 分支.

分支错了, 后续全部返工 — 所以这一步交给程序, 不交给模型.

LARK

"是否 bug 池"的判断写在第二步散文里: "Find the owner field — name matches 责任人 / 负责人 / 指派人 … AND type == "user"".

模型每次自己读字段列表去配, 没有结构校验脚本. 同一个表换个字段命名就可能错判.

02

身份 / 凭证解析

Identity resolution
PDF

不涉及身份, 跳过. 但同类 — 字段 ID 解析 — 完全由 extract_form_field_info.py 封装, 处理父子层级、checkbox/radio 合法值.

LARK

SKILL.md 用散文描述: "Parse data.user.open_id on success, or extract ou_[A-Za-z0-9_-]+ from a need_user_authorization error message."

lark-cli contact +get-user --as user 2>&1 # 模型每次自己挑这段两路解析 # 一旦错误消息文案变, regex 没人改
03

数据获取 + 分页

Fetch & paginate
PDF

无分页问题, 但 fill_fillable_fields.py 在写入前会再次拉取字段信息, 自校验 — 程序兜底.

LARK

用 --limit 200, 然后在文末加一句注脚: "如果 pool exceeds 200, records past the cutoff drop silently".

分页静默丢, 用户那边表现成"我的 bug 怎么少了". 这是教科书级别的应该脚本化的脆弱点.

04

写入前校验

Preflight
PDF

check_bounding_boxes.py 检查框相交、字体高度不够; fill_fillable_fields.py 校验 field_id 存在、checkbox/radio 值合法.

错误信息可行动, 直接指向哪个字段需要修.

LARK

+record-upsert 之前没有 preflight. 全靠 SKILL.md 的禁令清单 — "Don't update before user yes" — 由模型自觉.

没有机制确认 record_id 对应的就是用户刚选的 bug, 没有机制确认 状态 字段枚举值合法.

05

产出稳定

Stable output
PDF

产出是 PDF 文件本身. 格式由 fill_pdf_form_with_annotations.py 保证. 不存在"模型这次少写一句"的可能.

LARK

产出是一段 markdown 回复, 包含索引表 + 每条 bug 完整正文 + 收尾句. 模板写在 SKILL.md 散文里, 每次靠模型重新拼.

### ZB-XXXX — <标题> - **优先级 / 状态 / 类型 / 端 / 复现概率**: … - **前置条件**: … - **复现步骤**: <verbatim> - **实际结果** / **预期结果** / **复现环境** - **附件**: 2 张截图 / 日志 (未下载)

10-15 条 × 完整正文 — 每次模型从零拼一遍, 偶尔会漏字段或省略 — 而 SKILL.md 明确禁止省略.

Chapter 03

lark skill 的脆弱点

把上面五对差异落到 lark skill 自己, 一共能数出七个有概率出问题的位置. 按严重度排, 越靠前的越值得优先抽走.

P-01高

分页静默丢

--limit 200 之后, 超出的记录直接消失. SKILL.md 只提示"如果数量看起来异常, 告知用户可能超出 200" — 这是事后通知, 不是兜底.

触发场景: bug 池累计超过 200 条; 或多人同时上报后压栈.
P-02高

+record-upsert 无 preflight

用户说"改吧"之后, 模型直接拼 --record-id R --json '{"状态":"待验收"}'. 没有任何机制确认 R 就是用户刚才挑的那条, 也没有确认 状态 字段还接受"待验收"这个枚举.

触发场景: 用户在多条 bug 之间来回切; 表配置在生产中调整了状态枚举.
P-03高

身份解析靠 prose 描述正则

"parse data.user.open_id or extract ou_[…] from error message" — 这是一段需要分支处理的字符串解析, 写在散文里, 由模型每次自己实现.

触发场景: Lark 错误消息文案微调; 用户多账号切换.
P-04中

5 段 bash 由 AI 每次拼装

wiki resolve, field-list, get-user, record-list, record-upsert — 五段 lark-cli, 共享一组参数. 任何一段漏 --view-id 或 --format json 都会让后续步骤无声偏轨.

触发场景: 模型记忆漂移; MSYS_NO_PATHCONV=1 在 Windows git-bash 上忘加.
P-05中

回复模板靠记忆复刻

索引表 8 列 + 每条 bug 8 个文本字段 + 收尾句 — 全部以散文规范写在 Step 5. 模型重组 10-15 条时漏一项, SKILL.md 没有机器校验.

触发场景: 上下文紧张; 模型为节省 token 自行省略 "复现环境" 这类经常空的字段.
P-06中

字段定位无结构校验

owner / status / priority 三个字段的存在性靠模型读 +field-list 输出后自己判. 一旦字段被改名, 整个 skill 静默退化成"全部 bug 都展示".

触发场景: 表 schema 微调; 跨项目克隆到字段命名不一致的 bug 池.
P-07低

常量散落在 SKILL.md

WIKI_URL / WIKI_TOKEN / TABLE_ID / VIEW_ID 四个常量硬编码在 SKILL.md. "克隆到另一个仓库时改这四个值" — 这是配置, 应该独立成文件.

触发场景: skill 跨仓库复用; 多个 bug 池切换.
P-08低

用户确认的语义匹配在散文里

"改吧 / 改 / 确认 / 是 / yes / 好 / done" 列了 10 来个等价词. 这是 UX 约定, 但散落在工作流第 8 步里, 容易和其它确认场景混淆.

触发场景: 用户用了清单外的说法 ("行" "ok le"); 模型在 ambiguous 答复上仍上传.
停一下 · Exercise 02 ≈ 12 MIN · 先想再翻页

在看我的处方之前, 先开自己的 —

下一章是我给的处方, 但你的不一定要和我一致. 这一题的核心不是"猜对答案", 而是养成"看到痛点, 先想结构, 再想脚本"的肌肉记忆 — 这是优化任何 skill 时都用得上的判断顺序.

  1. 看完 P-01 到 P-08, 如果只能动 3 处, 你会动哪 3 处? 写出顺序, 并说明为什么是这个顺序 — 是按严重度、按依赖、还是按"修了能连带挡掉别的痛点"?
  2. 哪几个痛点可以被同一个脚本一次性挡住? 哪几个必须分开成不同的脚本? 想清楚分界, 你就理解了脚本的"职责颗粒度".
  3. 如果让你重新设计 lark skill 的目录, 你会画出什么样的树? 在纸上画一个 — 不需要写脚本名, 文件夹 + 大致用途就行. 等会儿和我的处方树并排放, 看看差在哪.
讲师提示这一题比 Exercise 01 难, 给到 10-12 分钟. 鼓励同事互相看一下彼此的树 — 不同人会有不同切法, 这个差异本身就是讨论素材. 收题时不要急着说"我的更对", 先问"你这样切的理由是什么", 然后再放出处方对照.

Chapter 04

建议的目标结构

沿用 pdf skill 的 inspect → transform → validate → execute 四段, 把上面的痛点逐一落到脚本和 reference. 下面是建议的文件树, 绿色 = 新增, 米色 = 保留并瘦身.

lark-bugs-assigned-to-me/ ├── SKILL.md ~3 KB — 入口 + DOT 流程图 + 脚本调度 ├── config.json 新增 — ZenBoard 四常量, 克隆改这里 ├── references/ │ ├── lark-cli-contract.md 新增 — 5 段 bash 的输入输出契约 │ ├── triage-template.md 新增 — markdown 回复模板 │ └── multi-bug-brainstorm.md 新增 — 4 问表 (按需加载) └── scripts/ ├── whoami.sh inspect — 当前用户 open_id ├── resolve_pool.sh inspect — URL/常量 → tokens ├── inspect_schema.py validate — 字段存在性校验 ├── fetch_my_bugs.py transform — 分页 + 过滤 + 排序 ├── format_triage.py transform — JSON → markdown ├── preflight_update.py validate — record_id + 状态枚举校验 ├── mark_pending_qa.sh execute — 状态改为 待验收 └── fetch_attachment.sh execute — 按需下载附件

whoami.shinspect

封装 +get-user --as user, 处理成功和 need_user_authorization 两路, stdout 只吐 open_id. 替换 P-03.

resolve_pool.shinspect

读 config.json 或解析传入 URL, 输出 base_token / table_id / view_id. 同一个入口, 两种来源.

inspect_schema.pyvalidate

跑 +field-list, 校验 owner/status/priority 都存在并定位 owner 的 positional index. 缺一就拒绝继续, 提示"不是 bug 池". 替换 P-06.

fetch_my_bugs.pytransform

自动分页直到耗尽, 用 open_id 过滤, 按 P0→P3 排序, 输出固定 schema 的 JSON. 替换 P-01 + P-04 的拼装风险.

format_triage.pytransform

读上一步 JSON, 吐出"索引表 + 完整正文 + 收尾句"的 markdown. 空字段写 — 而非省略 — 由脚本兜底, 不靠模型记忆. 替换 P-05.

preflight_update.pyvalidate

在 mark_pending_qa 之前检查: record_id 是否属于当前用户、状态枚举是否还接受"待验收"、是否会顺便覆盖其它字段. 替换 P-02.

mark_pending_qa.shexecute

极简包装, 单参数 --record-id. 不接受其它字段除非显式 --fix-summary / --pr-link. 拒绝静默写多余字段.

config.jsonconfig

四常量 + 默认 --limit + 排序优先级映射. 克隆 skill 到新仓库时只改这里. 替换 P-07.

效果 SKILL.md 从 15 KB → ~3 KB, 内容收敛为: 触发条件 / DOT 流程图 / 每一步指向哪个脚本或 reference / 用户确认 UX. 其余全部归位.

7项

脆弱点机器兜底

分页、身份、枚举、字段定位、报告格式、preflight、常量 — 七个原本依赖模型记忆的位置移交脚本.

1处

跨仓库克隆改动

克隆 skill 到新 bug 池, 只动 config.json. 当前需要在 SKILL.md 散文里找四处替换.

原则 这次优化不是把 lark skill 变得更复杂, 而是让它的复杂度从"模型每次都要记住"挪到"机器一次性写死". 判断力留在模型, 机械动作交给脚本 — 这是 pdf skill 给的样板, 也是这次重构唯一要复用的设计模式.
