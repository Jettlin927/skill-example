# Skill 设计原则 · 省流版

> 来源：<https://github.com/Jettlin927/skill-example>
> 适用于：写给 Claude Code / Codex 等 AI agent 用的 skill

## 一句话

**让 AI 判断该判断的，让脚本执行该执行的。SKILL.md 是路径，不是百科。**

## 三层结构

| 层 | 文件 | 干什么 |
|---|---|---|
| 入口 | `SKILL.md` | 列能力、给最常用代码、把复杂分支跳到子文档 |
| 深度 | `references/*.md` | 高级特性、疑难问题、稀少场景的详细资料 |
| 执行 | `scripts/*.py` 等 | 机械活：抽数据、转格式、校验、写入 |

## AI 用 skill 的四层思考路径

> 写 skill 时把这四层倒过来看，就能反推出每一层对应的设计原则。

1. **触发匹配** — AI 只看 `description`。所以 description 必须列穷触发动词，不是"这是 X 工具"。
2. **读 SKILL.md** — AI 在脑里建三张索引：能力清单 / 代码模板 / **跳转规则**。第三张最被低估。
3. **任务分流** — AI 把用户的自然语言对应到章节。所以章节标题就是任务名。
4. **深度执行** — 复杂分支下到子文档 + 脚本。AI 判断与脚本执行交替出现。

## 写 SKILL.md 的四条原则

1. **`description` 列穷触发动词** —— "读、合并、拆分、加水印、OCR……"。AI 用关键词匹配，动词越具体命中率越高。
2. **SKILL.md ≤ 500 行** —— 它是地图，不是百科。三类内容：能力清单 / 最常用代码模板 / 跳转规则（"If X, read Y.md"）。
3. **章节标题就是任务名** —— 用"合并 PDF / OCR / 加密"，不用"Section 1 / 2 / 3"。
4. **复杂分支拆到子文档 + 脚本** —— 一个分支超过 SKILL.md 半屏就该拆。子文档里再决定哪些步骤是脚本、哪些是 AI 判断。

## 六项自审清单

| # | 检查项 | 通过标准 |
|---|---|---|
| 1 | description 列穷动词了吗？ | 至少 3 个具体动词 |
| 2 | SKILL.md 多长？ | > 500 行就该考虑拆 |
| 3 | 有 `If X, read Y.md` 跳转吗？ | 复杂分支必须有 |
| 4 | 章节标题是任务名还是泛词？ | 用动词短语 |
| 5 | 有可以抽成脚本的机械步骤吗？ | 抽坐标 / 转格式 / 检查重叠都该是脚本 |
| 6 | 有没有把 AI 该判断的塞进脚本？ | 语义判断留给 AI |

## "AI 判断 + 脚本执行"的节奏

```
脚本 · 抽数据  →  AI · 判断含义  →  脚本 · 精确写入  →  脚本 · 校验输出
```

机械与语义交替出现。脚本输出是 AI 的输入；AI 决策是脚本的输入；脚本执行是落地。

- ❌ 别让 AI 临时写代码做机械活（坐标、格式、校验）——容易出错。
- ❌ 别在脚本里硬编码语义判断（"这个字段叫'生日'所以填用户的出生日"）——脚本做不了。

## 何时**不**该建 skill

- 一次性任务，写完就忘
- 完全没有可复用模式的随机请求
- AI 一句话就能搞定，且没有路径依赖
